## 1. 目的と要件

ダッシュボード(`web/`)を「朝開いたら3秒で今日やることがわかり、わからない言葉はその場で教えてくれる」体験に刷新する。

ユーザー要件:

1. 最高に使いやすいデザイン。検索も可能。
2. 今日「買い」「売り」の企業がパッと見でわかる。
3. 難しい用語はツールチップで初心者向けに噛み砕いて説明する。

ヒアリングで確定した方針:

- **対象範囲**: ホーム・成績・銘柄詳細の全3ページ刷新。
- **用語の見せ方**: やさしい言葉が主役(画面は「上がる確率 72%」、正式用語 prob_up はツールチップ内で教える)。
- **テーマ**: 現行ダーク系を磨く。赤=買い/上昇、青=売り/下落の日本株コンベンションは維持。
- **ホーム構成**: 案A「今日の結論ファースト」(ヒーロー→検索+全銘柄リスト→成績/配分→用語集)。

## 2. 共通基盤(全ページ横断)

### 2.1 用語辞書 + Term コンポーネント

- `web/src/lib/glossary.ts`: 用語辞書の単一ソース。

  ```ts
  type GlossaryEntry = {
    term: string      // やさしい表示名 例:「上がる確率」
    formal?: string   // 正式用語 例:「prob_up(予測上昇確率)」
    short: string     // 1〜2文の噛み砕き説明
    analogy?: string  // たとえ話(任意)
  }
  export const GLOSSARY: Record<string, GlossaryEntry>
  ```

- `web/src/components/Term.tsx`(client component): `<Term k="prob_up" />` で点線下線テキストを描画。
  - デスクトップ: hover/focus でポップオーバー表示。モバイル: タップでトグル、外側タップ/Escape で閉じる。
  - ポップオーバー内容: やさしい名 → 噛み砕き説明 → たとえ話(あれば) → 正式用語(小さく muted)。
  - a11y: `aria-describedby`、キーボードフォーカス可能。ビューポート端では位置をクランプ。
  - 外部ライブラリは使わず手書き(Radix 等の依存追加なし)。
- `web/src/components/GlossaryAccordion.tsx`: ホーム下部の「用語集」。GLOSSARY から自動生成し、説明の二重管理をなくす。
- 現行ホームの静的解説ブロック3つ(RSIって何?/出来高って何?/買いになる条件)は廃止し、ツールチップ+用語集+銘柄詳細の文脈表示に置き換える。

初期収録(約20語)。実装時にコピーを確定する:

| key | やさしい名 | 正式用語 | 説明の要点 |
|---|---|---|---|
| prob_up | 上がる確率 | prob_up(予測上昇確率) | 翌営業日に上がる見込み。70%=過去の似た局面で10回中7回 |
| gate | 成績テスト | KPIゲート(walk-forward OOS) | 過去データの模擬売買で基準を超えた銘柄だけ売買シグナルを出す試験。不合格→様子見 |
| rsi | 過熱感 | RSI(14) | 値上がりの勢い0-100。70超=買われすぎ、30未満=売られすぎ |
| volume_ratio | 商いの活発さ | 出来高/20日平均比 | 普段の何倍売買されているか |
| threshold | 判断ライン | シグナル閾値 | 何%を超えたら「買い」と言うかの銘柄ごとの基準(自動最適化) |
| volatility_guard | 値動きの荒さ制限 | ボラティリティガード | 荒すぎる銘柄に強い「買い」を出さない安全装置 |
| limit_price | 買ってよい上限 | 指値目安 | これより高値で追いかけないための目安 |
| stop_loss | 撤退ライン | 損切りライン | ここまで下がったら手仕舞いする目安 |
| drawdown | 一時的な落ち込み | ドローダウン | 直近ピークからの下落幅 |
| hit_rate | 的中率 | ヒット率 | 「買い」後に実際に上がった割合 |
| excess_return | 市場平均との差 | 超過リターン(対TOPIX) | 市場平均に勝った分 |
| topix | 市場平均(TOPIX) | TOPIX | 東証全体の平均的な値動き |
| calibration | 確率の正直さ | キャリブレーション | 70%と言った予測が本当に70%当たるか |
| brier | 予測のズレ点数 | Brierスコア | 確率予測の正確さ。0に近いほど良い |
| ic | 順位の当たり具合 | IC(情報係数) | 「上がりそう順」が実際と合っていたか |
| equity_curve | 資産の伸び | エクイティカーブ | シグナル通り売買した場合の資産推移(コスト込み) |
| ma | 平均線 | 移動平均線(MA5/20/60) | 直近N日の平均価格。トレンドの目安 |
| regime | 市場ムード | マクロレジーム | 金利・為替から見た相場の追い風/向かい風 |
| shadow_mode | 提案モード | シャドーモード | AI配分はまだ提案のみでシグナルに影響しない |
| cs_rank | AI順位 | クロスセクション順位 | 全銘柄を上がりそう順に並べた順位 |
| mae_mfe | 最大逆行/最大順行 | MAE/MFE | 保有中に最悪/最高でどこまで動いたか |

### 2.2 検索・絞り込み(StockExplorer)

- `web/src/components/StockExplorer.tsx`(client component)が検索文字列・フィルタ・ソートの状態を持ち、`dashboard_index.json` の全銘柄(約50)をクライアント内で即時フィルタする。
- 検索正規化: NFKC(全角/半角)+ 小文字化 + カタカナ→ひらがな折りたたみ。銘柄名とコード(`6701.JP` は「6701」でもヒット)の部分一致。
- フィルタチップ: すべて / 買い(BUY+MILD_BUY) / 売り(SELL+MILD_SELL) / 様子見(HOLD)。件数バッジ付き。
- ソート: おすすめ順(既定: 成績テスト合格の売買候補→確率の極端さ順)/ 上がる確率が高い順 / 低い順 / コード順 / 前日比順(データがある場合のみ表示)。
- URLクエリへの状態保存はしない(YAGNI。静的export での complexity 回避)。

### 2.3 色ルールの統一(シグナル・トークン)

- `web/src/lib/signal.ts` を表示トークンの単一ソースに拡張: アクション→{やさしいラベル, badge/card/text クラス, チャート用hex, アイコン}。
- **売り系を青に統一**: SELL=blue系(#3b82f6)、MILD_SELL=cyan系。現行の「売り=緑」表示は廃止。緑(emerald)は「成績テスト合格」など健全性の意味専用にする。
- 買い系: BUY=red系(#ef4444)、MILD_BUY=orange系(現行踏襲)。HOLD=slate。
- **`HOLD` の表示名を「様子見」に統一**(現行は英字 HOLD のまま)。表示はすべて signal.ts 経由にし、直書きクラスを排除。Recharts に渡す hex も同ファイルから import。
- 対象はフロントの表示のみ。LINE通知文言(`src/digest.py` 等)は本リデザインの対象外。

### 2.4 ナビゲーションとフッター

- 全ページ共通の薄いトップナビ: ホーム / 成績(+詳細ページは戻るリンク)。
- ヘッダー右に市場ムードのピル(RegimeBanner を縮約)。タップでポップオーバー: ひとこと要約+週次レポート/判断ログへのリンク(現行 RegimeBanner のリンクを移設)。
- フッター: 週次レポート・判断ログのリンク、免責の一文(投資判断は自己責任)。

## 3. ホーム(案A: 今日の結論ファースト)

上から:

1. **ヘッダー**: タイトル、最終更新日時、市場ムードピル。
2. **今日のAI判断(ヒーロー)** `TodayHero.tsx`:
   - 掲載条件: `latest_signal.gate_passed === true && action !== 'HOLD'`(LINE digest と同一ロジック)。
   - 買い候補(赤カード)と売り候補(青カード)。各チップ: 社名・コード・上がる確率、MILD は「やや買い/やや売り」小タグ。タップで詳細ページへ。
   - 並び: 買いは確率の高い順、売りは低い順。
   - 0件の日: 「今日は様子見の日 — 全銘柄、売買サインなし。それも判断のうち」と堂々と表示。
   - 見出しの日付はシグナルの date(取引日)を使う。ブラウザの今日と乖離する非取引日は「6/10(火)のAI判断」のように日付が語る。
   - 直下に「ほか N社は様子見」のミュート行。
3. **検索+全銘柄リスト** `StockExplorer.tsx`(2.2):
   - デスクトップ: テーブル風行(銘柄 / 株価+前日比 / 判断バッジ / 上がる確率ミニバー / 過熱感ラベル)。列ヘッダの「上がる確率」「過熱感」に Term ツールチップ。
   - モバイル: 2行コンパクトカードに組み替え。
   - 行全体が詳細ページへのリンク。買い行は薄赤、売り行は薄青の背景ティント。
4. **成績・配分のコンパクトカード**: 的中率・通算リターン+成績ページへのリンク。配分カードは現行 PortfolioCard を流用しスリム化+Term 適用(モード表示は「提案モード/自動反映」のやさしい名)。
5. **用語集アコーディオン** `GlossaryAccordion.tsx`。

## 4. 銘柄詳細ページ

1. **ヘッダー**: 社名・コード・現値・前日比(`data[]` 末尾2件から算出)・判断バッジ・「成績テスト合格/不合格」タグ。
2. **AIのひとこと** `SignalNarrative.tsx`: シグナルのフィールドから日本語文を自動合成。
   - 入力: `action, raw_action, prob_up, thresholds, gate_passed, confidence_reason, limit_price, stop_loss`。
   - 分岐(代表例):
     - 合格×買い系: 「判断は『買い』。上がる確率は74%で、この銘柄の買いライン(70%)を超えています。過去データでの成績テストにも合格している、本命のシグナルです。」
     - 不合格: 「AIの素の予測は『買い』寄り(74%)ですが、この銘柄は成績テスト不合格のため『様子見』にしています。」(confidence_reason を併記)
     - 純粋な様子見: 「確率52%で判断ラインの間。今日は様子見です。」
     - 売り系は鏡像。
   - `prob_up` や `thresholds` が null の場合は既存 `reason` 文字列の表示に劣化。
   - 買い系のときのみ「買うならこの値段まで(limit_price)/ここを下回ったら撤退(stop_loss)」の2チップを表示(値が無ければ非表示)。
3. **判断ラインゲージ** `ThresholdGauge.tsx`: 0–100%の横バーに 売り/やや買い/買いラインの目盛りと今日の確率マーカー。
4. **チャート**: 既存 StockChart を流用し、色を signal.ts トークンへ寄せる+MA/RSI トグルに Term 適用。既定レンジ3ヶ月。
5. **シグナル履歴**: 日付・バッジ・確率のタイムライン。`signal_outcomes_recent.json` に同一 ticker+entry_date の行があれば「結果 +2.1% ○」を併記(クライアント側 join、無ければ非表示)。

## 5. 成績ページ

1. **ヘッドライン**: 大きな数字で「5日後の的中率 58%」「市場平均との差 +N%」「通算 +6.2%」(5日ホライズンを主役に)。各語に Term。
2. **資産の伸び(対 市場平均)**: 既存 equity vs TOPIX チャート流用、凡例をやさしい名に。
3. **一時的な落ち込み**: 既存 drawdown チャート。
4. **確率の正直さ**: reliability チャート+「70%と言った時に本当に70%上がっているか」の一文。
5. **最近の結果一覧**: いつ/銘柄/判断/結果/市場平均との差/当たった?(○×)。MAE/MFE は Term 付き。
6. **モデルの健康診断**: model_quality を信号機表示(良好/注意。`drift_warning` を流用)。Brier/IC の生値は小さく表示し Term で解説。

## 6. データ契約・バックエンド変更

- **新規 JSON ファイルなし** → `daily-publish-dashboard.yml` の `--exclude` 変更は不要(`tests/test_publish_workflow.py` 影響なし)。
- 唯一のバックエンド変更: `src/dashboard.py` の index 出力(`dashboard_index.json` の各 `tickers.{code}` 直下)に `prev_close` と `change_pct` を**追加**(additive・optional)。前日比表示に使用。`latest_data` の dict には入れない(`records[-1]` は詳細 JSON の `data` 配列と同一オブジェクトのため)。
  - フィールドが無い旧 JSON ではフロントは前日比を非表示にして劣化運転(後方互換)。
  - `web/src/types/index.ts` に optional フィールドとして追記。
- 取得系は現行どおり `fetchJson.ts` 経由。バリデーション失敗・ファイル欠如はカード/セクション非表示(現行の安全設計を維持)。

## 7. 変えないこと(ガードレール)

- 日次パイプライン・シグナルロジック・通知・DB は変更しない(6章の additive export を除く)。
- Phase 2 シャドー原則、`tickers.yml` 不可侵、`docs/history_data.json` 廃止契約は不変。
- 新規 npm 依存ゼロ(ツールチップ・検索は手書き)。静的 export(`output: "export"` + basePath)維持。インタラクティブな新規コンポーネントはすべて client component。

## 9. リスクと対策

| リスク | 対策 |
|---|---|
| 売り=緑→青の色変更が既存チャートに波及 | 変更対象は「シグナル由来の色」のみと明記。signal.ts トークン経由に寄せ、株価チャートの陽線/陰線(赤/青)は現行維持 |
| HOLD→様子見の表記揺れ | 表示名は signal.ts の actionConfig 経由のみとし、コンポーネント内の直書きを排除 |
| ツールチップのモバイル挙動(端で見切れ・閉じられない) | 位置クランプ+外側タップ/Escape で閉じる。プレビューで実機確認 |
| 静的 export で useSearchParams 等の SSR 制約 | URL 状態は持たない(2.2)。新規コンポーネントは client component |
| `change_pct` 追加で旧 JSON との不整合 | optional 扱い+欠如時は前日比列ごと非表示 |
