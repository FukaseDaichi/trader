# ポートフォリオ最適化 拡張方針（全銘柄負荷を抑える）

作成日: 2026-03-01 (JST)

## 1. 結論（先に要点）

- 「全銘柄を毎日ダウンロードして、全銘柄を毎日学習」は重すぎるため非推奨。
- 実装方針は **3段階パイプライン** にする。
  - 広いユニバース（数千銘柄）: 低頻度更新
  - 候補銘柄（数百銘柄）: 軽量スコア
  - 最終採用銘柄（10〜40銘柄）: 現行の詳細モデル + 配分最適化
- まずは現行 `tickers.yml` の範囲でポートフォリオ最適化を入れ、その後候補母集団を拡張するのが安全。

## 2. なぜ全銘柄毎日フル解析が重いか

現行フロー（`main.py`）は銘柄ごとに以下を実施:

1. データ更新（`update_data`）: 銘柄ごとにHTTP取得
2. 特徴量生成
3. KPIゲート（OOS予測 + 閾値探索）
4. 予測モデル学習 (`train_and_predict`)
5. シグナル生成

現状でも1銘柄あたり処理量は小さくない。  
これを数千銘柄へ単純拡張すると、通信時間・学習時間・I/Oがボトルネック化する。

- 通信: 数千リクエスト/日
- 学習: 銘柄数に比例して線形増加
- ストレージI/O: parquet読み書きと特徴量計算が大量発生

したがって、**毎日フルユニバース再計算は避ける**のが基本方針。

## 3. 目標アーキテクチャ（銘柄単体 -> 全体順位付け + 配分）

## Layer A: Universe管理（低頻度）

- 対象: JP全銘柄候補（例: 3000〜4000）
- 更新頻度: 週次〜月次
- 役割:
  - 銘柄マスタ更新（上場廃止・新規上場）
  - 最低流動性・最低価格などの粗フィルタ
  - セクター情報の付与

## Layer B: 候補スコアリング（中頻度）

- 対象: Layer Aから絞った数百銘柄
- 更新頻度: 日次
- 役割:
  - 軽量特徴量（出来高、ボラ、短期モメンタム等）で予備ランキング
  - 上位N銘柄のみ Layer C へ送る

## Layer C: 詳細予測 + ポートフォリオ配分（高品質）

- 対象: 10〜40銘柄
- 更新頻度: 日次（配分更新は日次 or 週次）
- 役割:
  - 現行モデルの予測結果（`prob_up`）を使って期待値を推定
  - 銘柄横断で順位付け
  - リスク・相関・回転率制約を入れて最終ウェイト決定

## 4. 配分ロジック v1（実装容易な形）

複雑な最適化（厳密Q P）を最初から入れず、まずは再現性重視で以下を採用:

1. **採用候補条件**
   - KPIゲートを通過
   - `prob_up` が閾値以上（例: `mild_buy` 以上）
   - 流動性フィルタ通過

2. **銘柄スコア**
   - `edge = prob_up - mild_buy_threshold`
   - `risk_adj_score = edge / max(volatility, floor)`
   - これで銘柄横断ランキング

3. **ウェイト計算**
   - `w_raw = max(risk_adj_score, 0)`
   - 正規化して合計1にする
   - 上限制約（例: 1銘柄25%）
   - セクター上限制約（例: 40%）
   - 残余はキャッシュ

4. **回転率制御**
   - 前日比で変化が小さい場合は据え置き
   - 最小売買単位未満は発注しない

## 5. 「重さ」への具体対策

1. 日次は全銘柄学習しない（上位候補のみ学習）
2. ユニバース更新を週次/夜間ジョブに分離
3. データ更新を差分化（可能な範囲で増分取得）
4. キャッシュ前提で再計算範囲を限定
5. 処理時間のSLOを設定（例: 日次バッチ15分以内）

## 6. 段階的実装ロードマップ

## Phase 1（すぐ着手）

- 対象は現行 `tickers.yml` の有効銘柄のみ
- 銘柄横断ランキング + ウェイト計算を導入
- 出力:
  - `docs/portfolio_report.json`（採用銘柄、推奨ウェイト、キャッシュ比率、制約適用理由）

## Phase 2（次段階）

- 候補銘柄プールを 50〜200 に拡張
- 予備スコアリング層（Layer B）を追加
- 日次学習対象は上位Nに限定

## Phase 3（フル拡張）

- ユニバースを数千銘柄へ拡張（低頻度更新）
- セクター/相関制約を本格化
- 配分更新頻度と売買コスト最適化を調整

## 7. 既存コードに対する変更ポイント

- `main.py`
  - 銘柄単体 `signal` の後に、全銘柄集合を受ける `optimize_portfolio(signals, market_data)` を追加
- `src/`
  - `portfolio.py`（新規）: ランキング・制約付きウェイト計算
  - `universe.py`（新規）: 銘柄母集団管理（Phase 2以降）
  - `dashboard.py`（更新）: ポートフォリオ結果のJSON出力
- `web/`
  - 配分ウェイト、キャッシュ比率、セクター構成の表示（Phase 1では最低限で可）

## 8. 運用ルール（先に決める）

- リバランス頻度: 週次を基本、例外時のみ日次
- 1日最大売買回転率: 上限設定（過剰売買を防止）
- サーキットブレーク: 指標悪化時は新規エントリー停止
- 監視指標:
  - ポートフォリオCAGR, MaxDD, Sharpe, Turnover
  - 採用銘柄数、セクター集中度、キャッシュ比率

## 9. 最初の実装提案（この順で進める）

1. Phase 1として `portfolio.py` を追加（現行銘柄のみ対象）
2. `docs/portfolio_report.json` を出力
3. KPIゲートと整合する評価をバックテストへ追加
4. 効果確認後に候補銘柄プールを拡張（Phase 2）

## 補足回答（質問への直接回答）

- 「全銘柄ダウンロードして解析はかなり重くならない？」  
  -> **重くなる。** そのため、全銘柄を毎日フル処理する設計ではなく、段階的に絞る設計にするのが推奨。
