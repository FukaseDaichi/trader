# 株式収益システム 高収益化の開発方針（60点 -> 90点）

作成日: 2026-03-01 (JST)

## 0. 到達目標（定義を先に固定する）

- 目標は「予測精度」ではなく「手数料・スリッページ込みの資産曲線」を最大化すること
- 評価指標を以下に統一する
  - 年率リターン (CAGR)
  - 最大ドローダウン (MaxDD)
  - シャープレシオ / ソルティノレシオ
  - 1トレードあたり期待値 (Expectancy)
  - 売買回転率 (Turnover)

## 1. 現状60点の主因（現行実装ベース）

- 翌日上昇/下落の二値分類中心で、リターンの大きさを最適化できていない
- シグナル閾値が固定で、市況変化に追従しづらい
- 銘柄ごとの独立判定で、ポートフォリオ全体の資金配分最適化がない
- 取引コスト・スリッページ・出来高制約を含む収益評価が弱い
- ドリフト検知/再学習ルール/実験管理が不足し、改善サイクルが遅い

## 2. 開発方針（優先度順）

## P0: 先に作るべき「収益の土台」

1. **PnL直結の検証基盤を最優先で実装**
   - 取引コスト・スリッページ・約定制約込みのイベント駆動バックテストを標準化
   - Walk-forward で期間分割し、インサンプルとアウトオブサンプルを厳密分離
   - 成果条件: 指標を毎日自動出力し、ベンチマーク (TOPIX等) との比較が常時可能

2. **資金管理をモデルより先に強化**
   - 1銘柄あたり最大リスク、1日あたり最大損失、同一セクター上限を導入
   - ボラティリティ連動のポジションサイズ (例: ATR/日次ボラ連動) を採用
   - 成果条件: MaxDD と日次損失分布の悪化を抑制

3. **シグナル閾値を固定値から最適化へ移行**
   - 銘柄別・地合い別に確率キャリブレーション（Platt/Isotonic）
   - 閾値は「勝率最大化」ではなく「期待値最大化」で最適化
   - 成果条件: 取引あたり期待値の改善、無駄な売買回数の減少

4. **銘柄単体判定からポートフォリオ意思決定へ拡張**
   - 全銘柄を「期待リターン/リスク/相関」でランキングし、上位のみ採用
   - キャッシュ比率を動的制御（地合い悪化時は現金比率を上げる）
   - 成果条件: 連敗時の資産毀損スピードを低下

## P1: アルファ源泉を増やす

5. **ターゲット設計の高度化**
   - 二値分類に加えて「翌日期待リターン」「下方リスク」を予測
   - マルチタスク化して、方向性と利幅を同時に最適化

6. **特徴量をテクニカル偏重から拡張**
   - 市場レジーム: TOPIX/Nikkeiのトレンド、VIX代替ボラ指標、金利/為替
   - 企業イベント: 決算発表日、ガイダンス修正、配当/自社株買い情報
   - 流動性: 売買代金、出来高急増率、ギャップ継続性

7. **ユニバース管理を自動化**
   - `tickers.yml` を定期再評価し、流動性とトレンドで入替
   - セクター集中を制御し、相関が高すぎる銘柄を同時採用しない

8. **モデルを単一構成からアンサンブルへ**
   - LightGBM単体から、線形モデル/時系列モデル/ツリーモデルを併用
   - レジーム判定（上昇・下落・レンジ）でモデルを切替

## P2: 改善速度を上げる運用基盤

9. **実験管理を標準化**
   - 実験ID、特徴量セット、期間、コスト設定、結果指標を自動記録
   - 「何を変えて、何が効いたか」を追跡可能にする

10. **データ品質監視と障害耐性**
    - データ欠損、異常値、更新遅延の検知を毎日自動化
    - 異常時は売買停止またはロット縮小のフェイルセーフを実装

11. **ドリフト監視と再学習ポリシー**
    - 予測分布・特徴量分布・実績損益の乖離を検知
    - 「定期再学習 + 劣化時の臨時再学習」をルール化

12. **通知を「行動可能な意思決定情報」に進化**
    - BUY/SELLだけでなく、推奨ロット、想定RR、撤退条件を通知
    - ダッシュボードで「本日の採用理由」「除外理由」を可視化

## 3. 30/60/90日ロードマップ

1. **Day 0-30（P0完了）**
   - コスト込みバックテスト基盤
   - リスク上限とポジションサイジング
   - 閾値最適化の自動化

2. **Day 31-60（P1前半）**
   - 期待リターン予測の導入
   - 市場レジーム特徴量とイベント特徴量の追加
   - ポートフォリオランキング選定

3. **Day 61-90（P1後半 + P2着手）**
   - モデルアンサンブル/切替
   - 実験管理、ドリフト監視、再学習自動化
   - 通知/ダッシュボードの意思決定強化

## 4. 実装時の非機能ルール（利益を守るためのガード）

- 変更は必ず「同一期間・同一コスト条件」でA/B比較する
- 本番投入条件を事前定義する（例: OOSのExpectancyが一定以上）
- 複雑化より再現性を優先し、説明不能な優位性は採用しない
- 1回の改善で変数を増やしすぎず、因果を追える粒度で実験する

## 5. 最初に着手する具体タスク（次の1週間）

1. バックテストモジュールを追加（コスト・スリッページ・出来高制約）
2. `generate_signal` を期待値ベース閾値最適化方式へ変更
3. ポジションサイズ計算（銘柄別ボラ連動）を追加
4. 日次レポートに CAGR / MaxDD / Expectancy / Turnover を追記
